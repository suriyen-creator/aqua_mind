# AquaMind — Findings และจุดบอดของโครงการ

เอกสารนี้วิเคราะห์จาก `AquaMind_Final_Report.md` โดยแยกสิ่งที่รายงานกล่าวถึงแล้วออกจากสมมติฐานที่ยังไม่มีหลักฐานรองรับ ประเด็นสำคัญที่สุดคือ **ยังตอบไม่ได้ว่าระบบจะแม่นเพียงใดเมื่อไม่มีภาพดาวเทียม** เพราะยังไม่มีข้อมูลจริงและยังไม่ได้ทดสอบโมเดลแบบแยกตามโหมดข้อมูล การลดค่า Confidence ตามอายุภาพเป็นเพียงกลไกสื่อสาร ไม่ใช่หลักฐานว่า Probability ยัง Calibration ถูกต้อง

| จุดบอด | ทำไมเป็นปัญหา | วิธีแก้ที่แนะนำ |
|---|---|---|
| **ธุรกิจ — ยังไม่ชัดว่าใครคือผู้ใช้ ใครคือผู้ซื้อ และใครเป็นผู้อนุมัติงบประมาณ** | เกษตรกรอาจเป็นผู้ใช้ แต่ผู้จ่ายเงินจริงอาจเป็นสหกรณ์ หน่วยงานรัฐ ฟาร์มขนาดใหญ่ หรือบริษัทประกัน วงจรขายและความต้องการของแต่ละกลุ่มต่างกันมาก | ทำ Stakeholder map แยก User/Buyer/Payer/Approver และสัมภาษณ์อย่างน้อยกลุ่มละ 5–10 รายก่อนเลือกรูปแบบธุรกิจ |
| **ธุรกิจ — ปัญหาและความถี่ของปัญหายังอาศัยข้อมูลทุติยภูมิ** | ข่าวความเสียหายระดับพื้นที่ไม่ได้พิสูจน์ว่าเกษตรกรเป้าหมายแต่ละประเภทพบปัญหาบ่อยพอหรือจัดลำดับปัญหานี้สูงพอที่จะเปลี่ยนพฤติกรรม | ทำ Problem interview เก็บชนิดสัตว์ ระบบเลี้ยง ความถี่เหตุ ความเสียหาย วิธีรับข่าวปัจจุบัน และค่าใช้จ่ายในการรับมือ โดยไม่เสนอ Solution ในช่วงแรก |
| **ธุรกิจ — ยังไม่พิสูจน์ว่า Lead time 3–5 วันนำไปสู่การลดความเสียหายจริง** | การเตือนที่เร็วขึ้นไม่มีคุณค่าหากเกษตรกรไม่มีเครื่องให้อากาศ แรงงาน เงินทุน หรือสิทธิ์ย้ายกระชังเพื่อดำเนินการ | ทำ Actionability study วัดเวลาเตรียมอุปกรณ์ ความสามารถในการตอบสนอง ต้นทุนต่อ Action และผลลัพธ์ของ Action แต่ละชนิด |
| **ธุรกิจ — ยังไม่มีหลักฐาน Willingness to Pay** | SUS หรือความพึงพอใจไม่ได้แปลว่าจะจ่ายเงิน โดยเฉพาะเมื่อมีประกาศภาครัฐ กลุ่ม LINE และการสังเกตน้ำเป็นทางเลือกฟรี | ทดสอบราคาและข้อเสนอด้วย Landing page/Letter of Intent/Paid pilot แยกเกษตรกร สหกรณ์ หน่วยงานรัฐ และ B2B |
| **ธุรกิจ — ยังไม่มีการวิเคราะห์ทางเลือกและคู่แข่งตามงานที่ผู้ใช้ต้องทำ** | คู่แข่งไม่ใช่เฉพาะ Dashboard ดาวเทียม แต่รวมถึงประสบการณ์เกษตรกร เซนเซอร์ DO กลุ่ม LINE เจ้าหน้าที่ท้องถิ่น และการตรวจน้ำ | ทำ Competitive/Alternative matrix เทียบ Lead time, ความน่าเชื่อถือ, ความละเอียด, ต้นทุน, ผู้รับผิดชอบ และขั้นตอนหลังแจ้งเตือน |
| **ธุรกิจ — สมมติฐานว่าคำอธิบาย SHAP จะสร้างความเชื่อมั่นยังไม่ได้พิสูจน์** | ผู้ใช้อาจเชื่อจากการรับรองของเจ้าหน้าที่หรือผลตรวจ DO มากกว่ากราฟ AI และคำอธิบายมากเกินไปอาจทำให้สับสน | ทำ A/B usability test ระหว่าง Risk only, Risk+Data age, และ Risk+Explanation วัดความเข้าใจ การตัดสินใจ และ Over-trust |
| **ธุรกิจ — ยังไม่มีช่องทางเข้าถึงและดูแลผู้ใช้** | การติดตั้งแอปไม่ทำให้เกิด Adoption หากไม่มีผู้แนะนำ ฝึกใช้ ตอบปัญหา และยืนยัน Alert ในพื้นที่ | ทดลองช่องทางผ่านสหกรณ์/สำนักงานประมง/ผู้ขายอุปกรณ์ วัด Activation, Weekly active use, Alert acknowledgement และ Support load |
| **ธุรกิจ — การใช้งานอาจเป็นฤดูกาลและ Retention ต่ำ** | เหตุการณ์ Bloom ไม่ได้เกิดตลอดปี ผู้ใช้อาจลืมระบบ ปิดการแจ้งเตือน หรือไม่ยอมจ่ายรายเดือนต่อเนื่อง | วัด Retention ตามฤดูกาล พิจารณา Contract รายปีระดับหน่วยงานหรือเพิ่มคุณค่าคุณภาพน้ำที่ใช้ได้ตลอดปีโดยไม่ขยาย Scope ก่อนพิสูจน์ |
| **ธุรกิจ — ยังไม่มี Outcome metric** | Metric ปัจจุบันเน้น Recall/SUS แต่ไม่ตอบว่าระบบลดสัตว์ตาย ลดเวลาตรวจ หรือประหยัดค่าใช้จ่ายได้จริงหรือไม่ | กำหนด Business outcome เช่น Actionable lead time, ชั่วโมงที่ประหยัด, ค่าใช้จ่ายต่อฟาร์มที่หลีกเลี่ยงได้ และอัตรา Alert ที่นำไปสู่การตรวจยืนยัน |
| **เทคนิค/ดำเนินงาน — End-to-End ปัจจุบันยังแยกระหว่าง Model Demo กับ Live context** | Frontend–API–XGBoost–SHAP ทำงานครบเฉพาะข้อมูลสังเคราะห์ ส่วนบางแสนเชื่อมข้อมูลภายนอกได้แต่ยังไม่มี Sentinel-2 Feature ที่ตรงกับโมเดล จึงยังไม่พิสูจน์ Data จริง → Model → UI ครบเส้นทาง | คง Assessment gate ที่งดผลไว้ แล้วพัฒนา Sentinel-2/Cloud mask/NDCI pipeline ให้ได้ Feature schema เดียวกับโมเดล ก่อนทดสอบ Vertical slice ด้วย Ground truth และ Feedback ที่ยืนยันแล้ว |
| **เทคนิค/ดำเนินงาน — ยังไม่มี Service Level Objective และแผนเมื่อระบบล่ม** | ระบบแจ้งเตือนที่ล่มหรือข้อมูลค้างโดยไม่มีสถานะอาจอันตรายกว่าการไม่มีระบบ | กำหนด SLO ด้าน Freshness, Availability, Alert delivery, Recovery time เพิ่ม Health check, Monitoring, Incident runbook, Backup และข้อความ Degraded/Unavailable |
| **เทคนิค/ดำเนินงาน — พึ่งพา External services โดยไม่มี Exit plan** | GEE, Weather API, Copernicus, Map และ SMS อาจเปลี่ยนราคา Quota เงื่อนไข หรือหยุดให้บริการ | ทำ Data-provider matrix ระบุ SLA/Quota/License/Cost/Replacement, Cache ข้อมูลที่ได้รับอนุญาต และออกแบบ Adapter เพื่อเปลี่ยนผู้ให้บริการได้ |
| **เทคนิค/ดำเนินงาน — ยังไม่มีแผน Model lifecycle** | โมเดลอาจเสื่อมจากฤดูกาล ชนิดแพลงก์ตอน หรือพื้นที่ใหม่ และการอัปเดตโดยไม่มี Rollback อาจเพิ่มความเสี่ยง | ทำ Model registry, Versioning, Drift monitoring, Champion–challenger, Approval gate, Rollback และ Audit log |
| **เทคนิค/ดำเนินงาน — Last-mile alert ยังไม่ถูกทดสอบ** | เกษตรกรอาจมีอินเทอร์เน็ตไม่เสถียร ปิด Notification หรือรับ Alert ช้า | ทดสอบ LINE/SMS/โทรศัพท์/เจ้าหน้าที่เป็นหลายช่องทาง บันทึก Delivery/Acknowledgement และมี Escalation rule โดยคิดต้นทุนต่อข้อความ |
| **เทคนิค/ดำเนินงาน — Feedback จากผู้ใช้อาจไม่ใช่ Ground Truth** | ภาพสีน้ำและคำตอบผู้ใช้อาจผิดเวลา ผิดพิกัด ซ้ำ หรือ Bias และการให้ระบบเรียนรู้ตรง ๆ จะทำให้ Model drift ผิดทิศ | กำหนด Data provenance, Verification status, Duplicate check, Reviewer workflow และห้ามใช้ Unverified feedback ฝึกโมเดล |
| **เทคนิค/ดำเนินงาน — ความปลอดภัยและความเป็นส่วนตัวยังไม่มี Threat model** | พิกัดฟาร์ม เบอร์โทร และข้อมูลผลผลิตมีความอ่อนไหว การรั่วไหลกระทบธุรกิจผู้ใช้ | ทำ Data classification, Threat modeling, Least privilege, Encryption, Retention/deletion policy, Consent/Legal basis และ Security test ก่อน Pilot |
| **เทคนิค/ดำเนินงาน — ความรับผิดจากคำแนะนำยังไม่ถูกจัดการเชิงกระบวนการ** | Disclaimer อย่างเดียวอาจไม่พอเมื่อคำแนะนำทำให้เสียต้นทุนหรือสัตว์ตาย | ให้ผู้เชี่ยวชาญรับรอง Protocol, จำกัดคำแนะนำตามบริบท, เก็บ Audit trail, กำหนด Human escalation และประเมินความรับผิด/ประกันภัยก่อน Production |
| **เทคนิค/ดำเนินงาน — ยังไม่มี Capacity และ Cost test** | Raster processing หลายฟาร์ม, SHAP และ Alert พร้อมกันอาจเกิน Quota/งบ แม้ MVP จะทำงานได้ | ทำ Load model และ Load test ตามจำนวน AOI/ฟาร์ม/รอบประมวลผล วัด Cost per monitored km², per farm และ per alert |
| **Data gap — นิยาม Target label “Bloom” ยังไม่ชัด** | NDCI สูง, Chlorophyll-a สูง, น้ำเปลี่ยนสี, DO ต่ำ และ Harmful species ไม่ใช่เหตุการณ์เดียวกัน หาก Label ไม่ชัด โมเดลและคำเตือนจะตีความไม่ได้ | ทำ Label specification ระบุ Outcome, Threshold, วิธีตรวจ, พื้นที่, Lead window และระดับความรุนแรง แยก Bloom biomass, Harmful species และ Hypoxia หากข้อมูลรองรับ |
| **Data gap — Horizon 3–5 วันยังไม่มีวิธีสร้าง Label** | การใช้ค่าปัจจุบันเป็น Label ไม่ได้พิสูจน์ Early warning ระบบอาจเป็นเพียง Nowcast | สร้างตัวอย่าง ณ เวลา t และ Label ว่าเกิดเหตุใน t+1…t+5 หรือไม่ ป้องกันการใช้ข้อมูลอนาคต และรายงาน Metric แยกแต่ละ Horizon |
| **Data gap — จำนวน Ground Truth ยังไม่มี Power/Sample-size plan** | 10–15 จุดต่อ Campaign อาจให้เหตุการณ์จริงน้อยเกินไปสำหรับ Test set และ Confidence interval กว้างมาก | กำหนดจำนวน Independent events ขั้นต่ำจาก Precision ของ Metric ที่ต้องการ ทำ Multi-season sampling และร่วมใช้ข้อมูลหน่วยงานย้อนหลัง |
| **Data gap — ตัวแปรสำคัญยังขาด** | Nutrient/runoff, river discharge, tide/current, salinity, turbidity, bathymetry, water depth, farm type และ stocking density อาจมีผลต่อ Bloom/ความเสียหาย | จัดทำ Feature availability map แยก Must-have/Optional ตรวจ Abundance, Cost และ Leakage ก่อนเพิ่ม ใช้ Ablation test พิสูจน์คุณค่าแต่ละกลุ่ม |
| **Data gap — ปัญหา Optical coastal water ยังกล่าวไม่ครบ** | Sunglint, adjacency effect จากแผ่นดิน, bottom reflectance ในน้ำตื้น, ตะกอน และ Atmospheric correction ทำให้ NDCI ผิดได้ | เพิ่ม QC mask ชายฝั่ง/น้ำตื้น, Sunglint correction, Turbidity flag, Sensor/algorithm comparison และ Validation แยกตาม Optical water type |
| **Data gap — ความละเอียดต่างกันมากและเสี่ยง Pseudo-precision** | NDCI 20 m, SST ประมาณ 10 km และ Weather grid/API ไม่ได้แทนสภาพระดับกระชัง การ Resample ไม่เพิ่มข้อมูล | เก็บ Native resolution/footprint ใน Metadata, ห้ามแสดงเลขระดับฟาร์มจากข้อมูลหยาบ, ทำ Spatial aggregation ที่เหมาะสมและแสดง Source footprint บน UI |
| **Data gap — Temporal matching ยังไม่กำหนด Tolerance** | ตัวอย่างน้ำ ดาวเทียม อากาศ และน้ำขึ้นน้ำลงที่ต่างเวลากันอาจแทนคนละสภาพ | กำหนด Matching window ต่อชนิดข้อมูล บันทึก Timezone/Tide phase และวิเคราะห์ Sensitivity เมื่อเวลาห่างกัน |
| **Data gap — Sampling bias และ Coverage map ยังไม่มี** | จุดที่เข้าถึงง่ายหรือผู้ใช้สมัครใจอาจไม่แทนพื้นที่จริง ทำให้โมเดลแม่นเฉพาะบริเวณที่มีข้อมูล | สร้างแผนที่ Coverage ตามพื้นที่ ฤดูกาล Farm type และ Risk class ใช้ Stratified sampling และแสดงพื้นที่ Out-of-support |
| **Data gap — Data lineage และการเปลี่ยนเวอร์ชันต้นทางยังไม่ครบ** | การเปลี่ยน Processing baseline, API schema หรือวิธี QC อาจทำให้ Feature เปลี่ยนโดยไม่รู้ตัว | เก็บ Source product ID, processing version, checksum, ingestion time และ transformation code version พร้อม Reprocessing policy |
| **ไม่มีภาพ — ยังไม่มีหลักฐานว่าโมเดลจะยังแม่น** | Weather/SST บอกสภาวะเอื้อ แต่ไม่เห็นสถานะคลอโรฟิลล์ล่าสุด จึงอาจพลาด Bloom ที่เกิดเร็วหรือเหตุเฉพาะพื้นที่ | สร้างและทดสอบโมเดลแยก 2 ตัว: Full model และ No-image model แล้วประเมินแบบ Paired test บนวัน/พื้นที่เดียวกัน ไม่อนุมานจาก Full model |
| **ไม่มีภาพ — Confidence decay ยังไม่ใช่ Calibration** | การลดป้าย Confidence ตามจำนวนวันเป็นกฎเชิง UI ถ้าไม่เทียบเหตุจริง Probability อาจยัง Overconfident | ทำ Calibration แยกตาม `data_mode` และอายุภาพ รายงาน Brier/ECE/Calibration curve แล้วกำหนด Staleness gate จากผลจริง |
| **ไม่มีภาพ — การขาดภาพไม่ใช่การสุ่ม (Missing Not At Random)** | เมฆและฝนอาจเกิดพร้อมสภาพที่สัมพันธ์กับ Bloom ทำให้วันไม่มีภาพมี Distribution ต่างจากวันฟ้าใส | สร้าง Missingness indicators, ทดสอบ Metric เฉพาะช่วงเมฆ/มรสุม และทำ Stress test ห้ามเติมค่าด้วยวิธีที่ใช้ข้อมูลอนาคต |
| **ไม่มีภาพ — ยังไม่มีเกณฑ์ Go/No-go ของ Forecast-assisted** | หาก No-image model ต่ำกว่าเกณฑ์ ระบบยังอาจแสดงตัวเลขที่ผู้ใช้ตีความว่าพร้อมใช้งาน | กำหนดล่วงหน้าว่าแต่ละ Data-age bucket ต้องผ่าน Recall, Precision และ Calibration เท่าใด หากไม่ผ่านให้แสดง “ไม่สามารถคำนวณความเสี่ยง” และใช้ Field verification เท่านั้น |
| **ไม่มีภาพ — ไม่มีแผนสำรองสำหรับเหตุเปลี่ยนเร็วระดับฟาร์ม** | Regional SST/Weather ไม่สามารถเห็นเหตุเฉพาะจุด เช่น น้ำเสียไหลเข้า การให้อาหาร หรือกระแสน้ำเปลี่ยนทันที | ใช้ DO sensor/Manual check/รายงานเจ้าหน้าที่เป็น Local confirmation กำหนด Trigger ให้ตรวจภาคสนาม และพิจารณาแหล่งภาพเสริมเฉพาะเมื่อพิสูจน์ต้นทุนกับประโยชน์ |
| **การเงิน/ตลาด — ยังไม่มี Unit economics** | ตัวเลข Cloud อย่างเดียวไม่รวม SMS, Storage, Egress, Field sampling, Lab, Support, Sales, Compliance และ Model maintenance | สร้าง Cost model แยก Fixed/Variable คำนวณ Cost per farm/month, per km², per alert และ Gross margin ใน 3 ระดับการใช้งาน |
| **การเงิน/ตลาด — งบ Ground Truth ต่ำกว่าขอบเขต Validation** | การเก็บหลาย Campaign และตรวจ Chlorophyll-a/ชนิดแพลงก์ตอนมีต้นทุนสูงกว่าอุปกรณ์นำร่อง | ทำงบ Field validation เต็มรูป รวมแรงงาน เรือ เดินทาง ห้องแล็บ สอบเทียบ และ Contingency พร้อมระบุ Partner contribution ที่มีหลักฐาน |
| **การเงิน/ตลาด — รายได้ Premium API/ประกันภัยยังเป็นสมมติฐานไกลเกิน MVP** | ลูกค้ากลุ่มประกันต้องการข้อมูลที่ตรวจสอบย้อนกลับ ความเสถียร และหลักฐานเชิงสถิติสูงกว่าระบบ Demo | แยก Roadmap Research → Operational pilot → Paid institutional service; อย่าใช้ประกันภัยเป็นฐานประมาณรายได้จนผ่าน Multi-season validation และ Legal review |
| **การเงิน/ตลาด — ไม่มี CAC, Sales cycle และ Procurement risk** | การขายให้รัฐหรือองค์กรอาจใช้เวลานาน มีค่า Integrate/อบรม/จัดซื้อ และรายรับไม่สม่ำเสมอ | ทำ Pilot funnel บันทึก Cost/time ตั้งแต่ Lead ถึง Contract สร้าง Scenario cash flow แบบ Best/Base/Worst และวาง Runway |
| **การเงิน/ตลาด — เสี่ยงถูกบริการสาธารณะหรือโครงการวิจัยทดแทน** | หากหน่วยงานรัฐเปิดข้อมูลฟรี ผู้ใช้ไม่จำเป็นต้องจ่ายค่า Dashboard ทั่วไป | วางคุณค่าที่แตกต่างบน Workflow, Farm configuration, Verified alert, Integration และ Support พร้อมทดสอบว่าลูกค้ายอมจ่ายส่วนใด |
| **การเงิน/ตลาด — License/Terms และความรับผิดอาจเพิ่มต้นทุน** | การใช้ข้อมูล แผนที่ API และ Cloud เพื่อการค้าอาจมีเงื่อนไขต่างจากงานวิจัย และความเสียหายจาก Alert อาจมีต้นทุนทางกฎหมาย | ตรวจ Terms ต่อแหล่งข้อมูลและ Dependency ก่อนวางราคา ทำ Legal/PDPA review และตั้งงบ Security, Compliance, Contract และ Insurance |
| **การเงิน/ตลาด — ไม่มีแผนเงินทุนหลังทุนประกวด** | การดูแลข้อมูลและระบบเตือนภัยเป็นภาระต่อเนื่อง หากทีมหมดทุนระบบอาจหยุดในช่วงที่ผู้ใช้พึ่งพา | กำหนด Owner, Maintenance budget, Support hours, Sunset policy และ Funding milestones ก่อนรับผู้ใช้จริง |

## ข้อสรุปเรื่อง “ถ้าไม่มีข้อมูลภาพ ยังจะแม่นอยู่หรือไม่”

**คำตอบจากหลักฐานปัจจุบันคือ: ยังไม่ทราบ และห้ามถือว่ายังแม่น** ระบบ MVP ไม่มีภาพจริงและไม่มีการทดสอบ No-image mode ข้อมูลอากาศกับ SST ระดับภูมิภาคอาจช่วยบอก “สภาวะเอื้อ” แต่ไม่ยืนยันว่าบริเวณฟาร์มมีแพลงก์ตอนเพิ่มขึ้นแล้ว ดังนั้น V2 ควรกำหนดให้ Forecast-assisted เป็นโมเดลแยก มีการ Calibration แยก และมีเกณฑ์หยุดแสดงตัวเลขเมื่อไม่ผ่าน Validation หากไม่มีภาพและไม่มีข้อมูลภาคสนามที่เชื่อถือได้ ระบบควรแจ้งว่า **ไม่สามารถประเมินระดับฟาร์มได้ในขณะนี้** แทนการแสดงค่าความเสี่ยงที่ดูแม่นยำเกินหลักฐาน
